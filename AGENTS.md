#### Co-op smoke checklist (2 clients, same DLLs/cfg)

1. Demand: same high-demand products; lists feel consistent
2. MultiBoxCarry: only local queue; partner cannot take queued boxes

4. Training: guest sees host levels after join; host train updates guest

## Stack facts (v1.4.2)

- **IL2CPP** Unity game → use BepInEx 6 Unity.IL2CPP + HarmonyX.
- Game APIs are **Il2Cpp** wrappers: `Il2CppSystem.Collections.Generic.List<T>`, not `System.Collections.Generic.List<T>`, for game objects.
- Coroutines from managed code need `.WrapToIl2Cpp()` (`BepInEx.Unity.IL2CPP.Utils.Collections`).
- Injected MonoBehaviours need `ClassInjector.RegisterTypeInIl2Cpp<T>()` before use (see `Plugin.RegisterIl2CppTypes`).
- Harmony patches are applied per type via `CreateClassProcessor` in `Plugin.ApplyPatches` — **one bad patch method can fail the whole patch class**. Prefer small patch classes / correct parameter names.

## Critical 1.4.2 API changes (learned the hard way)

### Restockers are `Clerk`, not legacy `Restocker`

- Hire/spawn/active lists use **Clerk** APIs.
- Do **not** inject/patch `___m_ActiveRestockers` as if Restocker still owned stocking.
- Legacy `RestockerLogic` IEnumerator patches often **mismatch Il2Cpp iterators** → disable or avoid.
- Working XP path: `ClerkRestockerXpPatch` on `RestockDisplay.OnEnd`, `Clerk.PlaceBox_Network`, etc.

### Baker

- Spawn/despawn: `BakeryManager.SpawnBaker` / `DespawnBaker`.
- Hire IDs: `EmployeeManager.m_BakersData`.
- Tunables that work: `m_Agent` speed, `m_PlacingProductsInterval`.
- **Do not** train oven bake time (`Oven.BakeDuration`) — oven-owned.
- `m_BoxTransitionTime` writes caused trouble; avoid until defaults are measured.
- Early `AgentAngularSpeed = Max(0, Rapidity-2)*240` → **0** at low levels breaks LookAt / frozen grab. Floor angular/accel (e.g. ≥120 / ≥8).
- XP: prefer **local** methods (`TakeProducctFromOven` typo is real, `PickUpBox`) — `_Network` alone often never fires in SP.
- Bad XP hook: `CollectBakedProductState.Exit` (fires on every aborted FSM transition → phantom XP).

### Ice Cream Helper

- Hire/spawn via `EmployeeManager` ice cream lists (`m_ActiveIceCreamHelpers`, etc.).
- Tunable: `m_CurrentActionDelayInterval` (serve delay), optionally animation speed lists.
- **XP success only:** `Customer.DeliverIceCream` — capture helper in **Prefix** (stand is cleared before Postfix). Do **not** grant on `EndWork` (fires on missing-cone complaints too).

### Customer Helper (self-checkout)

- Vanilla walk + scan must stay intact. **Do not** Prefix-replace `StartCustomerHelperCheckout` / `StartCustomerHelperCheckoutOrder` with custom scan-only coroutines (skips walk → stand in place while SCO animates).
- Current approach: Postfix those methods → `ApplyRapidity` + `WatchHelpAndGrantExp` (+3 XP when help/scan ends).
- Parameter name for Order patch must match game: `CustomerHelper helper` (not `customerHelper`) or HarmonyX fails the whole class.
- Il2Cpp **properties** (`m_CustomerHelperScanIntervals`, boost level) are not always real fields — use direct property access, not `PrivateFld` reflection.

### Cashier / SCO

- Custom cashier scanning is separate from Customer Helper.
- `Checkout.Cashier` / `ProductScanned(..., cashier: true)` can NRE on self-checkout (no cashier). Guard with try/catch; don’t assume Cashier exists.

### Security / Janitor

- Security XP: detect / catch / restock stolen product hooks in `SecurityLogic`.
- Janitor: mop/dust/garbage broadcast-style hooks; keep grants modest (≈2).

## Employee Training Program architecture

### Plugin entry

- `EmployeeTraining.Plugin` — BepInEx plugin id `jp.tsuteto.sms.EmployeeTrainingProgram`.
- Loads localization, registers Il2Cpp UI types, applies all Harmony patches, wires scene/locale events.
- Save data: `ETSaveManager` under a CashierTraining-style folder (legacy name).

### Per-role pattern

Each role usually has:

| Piece | Role |
|-------|------|
| `*Skill` / `*SkillTier` / `*SkillData` | Level table, wage, derived stats |
| `*SkillManager` | Assign/spawn/despawn/sync Training App |
| `*Logic` | ApplyRapidity / GiveExp / role behavior |
| `*Patch` / `*XpPatch` | Harmony hooks |
| `*AppTab` / `*TrainingProgressItem` | PC Training App UI |

Roles: Cashier, Restocker (Clerk), CsHelper, Security, Janitor, Baker, IceCreamHelper.

### Training App

- Built onto the in-game PC (`PCTrainingApp`).
- Tabs composed at runtime from employee panels.
- Localization keys in `LocalizationSerializer.Default()`; missing keys fall back to English.

### Skill application rules

- Prefer **Postfix** on vanilla boost setters (`SetBakerBoost`, `SetHelperBoost`, `SetCustomerHelperBoost`, `SetRestockerBoost`, `SetJanitorBoost`) then apply **training × relative boot** (`BoostStacking`: current vanilla tier / tier 0). Boots stack on top of training; never replace one with the other.
- Never zero NavMeshAgent angular speed.
- When replacing a vanilla coroutine: you own movement + completion cleanup; if unsure, **don’t replace** — observe + adjust fields + grant XP on success.

### XP design principles

1. Grant on **successful work evidence**, not FSM `Exit` / cleanup.
2. Debounce duplicate network/local double-fires.
3. Early game should stay slow (lvl2 ≈ 50 XP on shared tables).
4. Prefer finish bonuses over per-item spam (especially Customer Helper).
5. `GiveExp` should use `GetSkill` first; avoid re-`AssignSkill` every grant (log spam / UI churn).

Rough current pacing (subject to tuning):

| Role | Grant style |
|------|-------------|
| Restocker | ~2 stock display, ~1 place box |
| Baker | 1 on real take/place/pickup |
| Ice Cream | 1 on deliver only |
| Cashier | 1 scan, 2 checkout finish |
| CsHelper | 3 help-finished only |
| Janitor | ~2 per clean action |
| Security | detect/catch/restock product |

## Harmony / Il2Cpp pitfalls checklist

- [ ] Patch parameter names match interop method parameters exactly.
- [ ] One failing method can kill the entire `[HarmonyPatch]` class — split risky patches.
- [ ] Don’t use `PrivateFld` on Il2Cpp auto-properties.
- [ ] IEnumerator patches against Il2Cpp state machines are fragile; prefer Postfix on public methods.
- [ ] After changing patches, check `LogOutput.log` for `Failed to apply patches` and NREs.
- [ ] Singleplayer may call local methods only — patch both local and `_Network` when needed.
- [ ] Capture values in Prefix if Postfix runs after the game clears them.

## Debugging workflow

1. Reproduce in-game.
2. Read tail of `BepInEx\LogOutput.log` for `EmployeeTrainingProgram` / HarmonyX errors.
3. Confirm the intended patch class applied.
4. If AI “stands still”, check whether a Prefix returns `false` and skips vanilla movement.
5. If XP missing, verify the hooked method is actually invoked (add temporary log, or ilspy call sites).
6. If XP floods, the hook is probably an Enter/Exit/cleanup, not a success action.

## Adding a new employee role (template)

1. New folder `EmployeeTraining.EmployeeFoo/`.
2. Implement Skill + Manager + Logic + Patch + AppTab + ProgressItem.
3. Register save list in `ETSaveManager` / `TrainingData`.
4. Wire tab in `PCTrainingApp`.
5. Register any Il2Cpp UI type in `Plugin.RegisterIl2CppTypes`.
6. Sync on scene load like Baker/IceCream managers.
7. Pick XP hooks from real success methods; keep early grants low.
8. Build, install, verify patches in log.

## What not to do

- Don’t commit secrets or whole game installs.
- Don’t force-push or amend unless explicitly asked.
- Don’t reintroduce Collect/EndWork-as-success XP hooks.
- Don’t Prefix-replace Customer Helper checkout start just to change scan speed.
- Don’t treat Restocker legacy coroutines as the 1.4.2 restock path.
